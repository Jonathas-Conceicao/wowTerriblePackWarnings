### v0.1.1 Adding Mob Categories (In Progress)

**Milestone Goal:** Per-mob category system (boss/miniboss/caster/warrior/rogue/trivial/unknown) with runtime detection so ability alerts filter correctly by mob type. Skyreach fully categorized; all other dungeons default to unknown.

#### Phase Checklist

- [x] **Phase 19: Data Layer** - Define mobCategory field schema and populate all dungeon data files (completed 2026-03-23)
- [x] **Phase 20: Runtime Detection and Scanner Matching** - Cache classification APIs per nameplate, derive category, apply filter with wildcard logic (completed 2026-03-23)
- [x] **Phase 21: Config Display** - Read-only color-coded category tags in mob header rows and category-aware search (completed 2026-03-24)
- [x] **Phase 22: Dungeon Category Index** - Apply categories to 7 remaining dungeons, remove isBoss, update PackFrame boss detection (completed 2026-03-24)

## Phase Details

### Phase 19: Data Layer
**Goal**: Every AbilityDB entry carries a settled mobCategory field with a clear vocabulary, Skyreach fully categorized and all other dungeons explicitly defaulting to unknown
**Depends on**: Nothing (first phase of milestone)
**Requirements**: DATA-01, DATA-02, DATA-03, DATA-04
**Success Criteria** (what must be TRUE):
  1. Every npcID entry in every Data/*.lua file has a `mobCategory` field — no entry is nil at runtime
  2. All 22 Skyreach mobs have the correct category from the provided reference table, not the default
  3. All mobs in the other 8 dungeons have `mobCategory = "unknown"` as an explicit string assignment
  4. `mobClass` (uppercase WoW token e.g. `"WARRIOR"`) and `mobCategory` (lowercase role string e.g. `"warrior"`) are named and commented such that no code can confuse them
**Plans:** 2/2 plans complete

Plans:
- [ ] 19-01-PLAN.md — Skyreach mobCategory migration with per-mob categories and Outcast Servant stub
- [ ] 19-02-PLAN.md — Non-Skyreach dungeons mobCategory = "unknown" migration (7 files)

### Phase 20: Runtime Detection and Scanner Matching
**Goal**: NameplateScanner derives a runtime category per mob at nameplate-add time, keeps it current via event, and gates ability activation so only matching-category (or unknown-wildcard) abilities fire
**Depends on**: Phase 19
**Requirements**: DETC-01, DETC-02, DETC-03, DETC-04, DETC-05, SCAN-01, SCAN-02, SCAN-03
**Success Criteria** (what must be TRUE):
  1. In a Skyreach dungeon run, mobs with a non-matching category do not trigger their abilities in the display
  2. In any non-Skyreach dungeon run, all abilities display normally — the unknown wildcard suppresses no alerts
  3. `UnitClassification` and `UnitIsLieutenant` are cached at `NAME_PLATE_UNIT_ADDED` and updated by `UNIT_CLASSIFICATION_CHANGED`, never called inside `Tick()`
  4. `UnitIsLieutenant` is wrapped in `pcall` — the addon does not error if the API misbehaves
  5. Scanner reads `mobCategory` from `ns.AbilityDB[npcID]` at match time with no duplication into SavedVariables
**Plans:** 1/1 plans complete

Plans:
- [ ] 20-01-PLAN.md — Pipeline mobCategory propagation + full NameplateScanner category rework + Core.lua event registration

### Phase 21: Config Display
**Goal**: Config window mob header rows show a read-only color-coded category tag, and category search terms return matching mobs
**Depends on**: Phase 19
**Requirements**: UI-01, UI-02, UI-03
**Success Criteria** (what must be TRUE):
  1. Each mob header row in the config tree shows a color-coded bracket tag (e.g. `[Caster]`, `[Boss]`) with no editable control beside it
  2. Searching "boss", "miniboss", "caster", "warrior", "rogue", or "trivial" in the config search box returns mobs whose mobCategory matches
  3. Searching "unknown" returns mobs without a specific category assignment
**Plans:** 1/1 plans complete

Plans:
- [ ] 21-01-PLAN.md — Color-coded category tags in mob headers and category-aware search filter

### Phase 22: Dungeon Category Index
**Goal:** Categorize all mobs in the remaining 7 dungeons with correct mobCategory values from Data/MobCategories.md, remove isBoss field from DungeonEnemies.lua (replaced by mobCategory = "boss" in AbilityDB), and update PackFrame.lua boss pull detection to use AbilityDB categories
**Depends on:** Phase 21
**Requirements**: CAT-01, CAT-02, CAT-03, CAT-04, CAT-05, CAT-06
**Success Criteria** (what must be TRUE):
  1. Every AbilityDB entry in all 7 dungeon data files has its correct mobCategory from MobCategories.md
  2. No isBoss field exists anywhere in DungeonEnemies.lua
  3. PackFrame boss detection uses ns.AbilityDB[npcID].mobCategory == "boss" instead of npcIdIsBoss
  4. Mindless Laborer (252557) present in both DungeonEnemies and PitOfSaron.lua
  5. All 9 coverage-gap stubs added for full AbilityDB coverage
**Plans:** 2/2 plans complete

Plans:
- [ ] 22-01-PLAN.md — Apply correct mob categories to all 7 dungeon data files + add 9 coverage stubs
- [ ] 22-02-PLAN.md — Remove isBoss from DungeonEnemies, add Mindless Laborer, update PackFrame boss detection
