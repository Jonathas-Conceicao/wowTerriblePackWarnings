```lua
local DUNGEON_NAMES = {
    windrunner_spire = "Windrunner Spire",
    imported = "Imported Route",

```

This ensures the PackFrame header shows "Imported Route" instead of the raw key "imported" when displaying imported packs.

Additionally, after the existing `for dungeonKey in pairs(ns.PackDatabase)` initialization block at the bottom of the file (lines 218-220), add auto-expansion for the "imported" key:
```lua
-- Ensure imported route is expanded when restored
expandedDungeons["imported"] = true
```

Actually, the existing code already expands ALL dungeons by default (line 218-220 iterates all PackDatabase keys). Since the imported key is added at runtime (on restore), we need to ensure it gets expanded. The simplest fix: in the PopulateList function, auto-expand any dungeon key that doesn't have an explicit collapsed state. The current code already does this implicitly via `expandedDungeons[dungeonKey]` being nil (falsy) — so new dungeons start collapsed.

Better approach: In the Refresh function, before calling PopulateList, auto-expand any new dungeon keys:
```lua
function PackUI:Refresh()
    -- Auto-expand newly added dungeon keys
    for dungeonKey in pairs(ns.PackDatabase) do
        if expandedDungeons[dungeonKey] == nil then
            expandedDungeons[dungeonKey] = true
        end
    end
    PopulateList()
end
```

This ensures imported packs are visible immediately after import or restore without requiring the user to click to expand.
